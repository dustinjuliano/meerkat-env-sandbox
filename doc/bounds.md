# Bounds and Allocation Safeguards Analysis

This document details the boundary conditions, integer overflow vulnerabilities, and resource limits of the current `Context` block allocation system.

## 1. Diagnostic Analysis of Flaws and Vulnerabilities

The current implementation lacks safeguards on index calculations and identifier increments, introducing vulnerabilities:

### Arithmetic Overflow in Region Bounds

In [`src/env.rs`](../src/env.rs), the allocation frontier is advanced using raw addition:
`let end = BlockId(begin.0 + size)`

- **Debug Mode Assertion Panic**: In debug compilations, Rust inserts runtime overflow checks on standard arithmetic operators. If `begin.0 + size` exceeds `u32::MAX`, a thread panic occurs, halting execution immediately
- **Release Mode Silent Wraparound**: In release builds, overflow checks are disabled by default. The addition wraps around modulo `2^32`. If the wrap yields a small value, it overlaps with existing allocations or the sentinel `BlockId(0)`
- **Sentinel Collision and Infinite Loops**: A wrapped `BlockId` points to the sentinel or active memory, corrupting LCRS links (`up`, `down`, `next`). Traversal operations like walking parent scopes enter infinite lookup loops

### Monotonic ID Exhaustion (Counter Leak)

The allocation counter `block_next_id` is monotonic and advances permanently on a freelist miss:

- **Unreclaimed Space Expansion**: Even if all active `Region` handles are freed, the counter is never decremented. Re-allocating a region that cannot be satisfied by the exact size slots of the `block_freelist` forces the counter forward
- **Artificial Exhaustion**: In a long-running REPL session or event loop executing millions of incremental code updates, the counter will eventually hit the `u32::MAX` limit even if active memory consumption remains near zero
- **Permanent Allocator Block**: Once the limit is breached, the allocator is permanently locked, rendering the system unable to process further allocations regardless of actual physical RAM availability

### Indexing and Boundary Dereferences

The internal data structures do not validate block indices during resolution:

- **Missing Index Guard**: Resolving structural relationships relies on raw index lookups like `self.blocks[id.0 as usize - 1]`
- **Cross-Context Pointer Corruption**: If a `BlockId` generated by one `Context` is resolved in another `Context`, the lookup executes without validation, returning corrupted structure data or panicking due to out-of-bounds index references

## 2. Memory Capacity vs ID Limits

There is a misalignment between physical memory capacity and the `u32` identifier space:

- **Memory Consumption**: A `blocks` vector of size `u32::MAX` containing 12-byte `Block` elements requires approximately `51.5 GB` of contiguous physical RAM
- **Preemptive OOM**: In practice, a standard user virtual machine will encounter a heap allocation Out-Of-Memory (OOM) panic from the system allocator long before reaching the theoretical `u32` limit
- **No Compaction**: Because the backing store is a single flat `Vec<Block>`, memory cannot be reclaimed or shrunk unless we implement a compaction pass that shifts active regions and rewrites all external references

## 3. Recommended Mitigation Strategies

To secure the allocator against crashes and undefined behavior under extreme allocations, several strategies can be proposed:

### Safe Checked Arithmetic and Error Propagation

Replace raw increments with safeChecked operations and bubble up allocation failures to callers:

- **Checked Calculations**: Rewrite the allocation step using `begin.0.checked_add(size)` and verify the resulting boundary does not exceed `MAX_BLOCK_ID`
- **Signature Refactoring**: Refactor `region_alloc` to return `Result<Rc<Region>, AllocError>` or `Option<Rc<Region>>` instead of panicking on overflow
- **Caller Recovery**: Allow the compiler or virtual machine interpreter to catch allocation limits, flush transient resources, and report errors gracefully instead of crashing the process

### Two-Pass Copying Compaction

Introduce an active compaction phase to recycle ID gaps and shrink the backing vector:

- **Mark Phase**: Traverse active `Region` handles to identify all occupied block indexes in the backing store
- **Relocate Phase**: Copy active blocks to the front of `blocks`, reducing the vector size and resetting `block_next_id` to the end of the compacted range
- **Pointer Rewriting**: Update `up`, `down`, and `next` indices inside the relocated blocks by applying the relocation offset, and rewrite the active `Region` boundaries held by compiler/VM clients

### Generational Local Allocators

Isolate scope allocation lifetimes by splitting the global context into localized scopes:

- **Epoch-Based Contexts**: Allocate separate `Context` engines for localized compiler jobs (e.g. compiling a single module or service block)
- **Complete Environment Disposal**: Once compilation or execution of a scope is complete, the entire local `Context` is dropped, recycling its `u32` address space completely without exhausting global VM resources
- **Immutable Parent Linking**: Local contexts can share read-only access to a global parent scope tree, keeping local mutations clean and isolated
